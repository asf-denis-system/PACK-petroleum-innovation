# [UP.M.016] Управление конфигурацией инновационного проекта

## Определение

Метод отслеживания согласованности всех аспектов инновационного проекта на каждом этапе его жизненного цикла. Основан на принципах IEC 81346 (многоаспектное описание объекта) и DevOps (непрерывная интеграция, управление конфигурацией).

## Мотивация

Инновационные проекты в нефтянке буксуют не из-за технологии, а из-за рассогласования: обоснование сделано на устаревшей модели, скважина-кандидат в ремонте, бюджет перераспределён. Это конфигурационные коллизии — те же, что в IEC 81346 для заводов, но применительно к проектам.

## Аспекты инновационного проекта

| Аспект | Аналог IEC 81346 | Что отслеживать | Кто видит |
|--------|------------------|-----------------|-----------|
| Проблема | Функция `=` | Какую проблему решаем? Актуальна ли? | Геолог, технолог |
| Технология | Продукт `-` | Какой метод/оборудование? Версия? | Инноватор, поставщик |
| Объект пилота | Место `+` | Какие скважины/участки? Доступны ли? | Промысел |
| Тип решения | Тип `%` | Класс ГТМ для сравнения | Аналитик |
| Ресурсы | Другой `#` | Бюджет, люди, сроки | Руководство |

## Конфигурационные единицы проекта

Для каждого инновационного проекта фиксировать:
1. **Версия модели** — на какой версии ГДМ/геомодели обосновано
2. **Список скважин-кандидатов** — с текущим статусом каждой
3. **Версия технологии** — если методика менялась
4. **Обоснование** — привязанное к версии модели
5. **Бюджет** — выделен / перераспределён / заморожен
6. **Исполнитель** — назначен / доступен

## Проверка конфигурационных коллизий (чеклист)

Перед каждым переходом состояния проекта:
- [ ] Модель, на которой обосновано, актуальна? (не устарела за время согласования)
- [ ] Скважина-кандидат доступна? (не в ремонте, не переведена)
- [ ] Бюджет выделен? (не перераспределён)
- [ ] Исполнитель назначен и доступен?
- [ ] Технология не изменилась с момента обоснования?

## Мантра M.003 для инноваций

«Начинать с управления конфигурацией» применительно к BV-TV:
1. **Прекрати делать ненужное** — закрой проекты с устаревшими обоснованиями
2. **Наведи порядок** — привяжи каждое обоснование к версии модели
3. **Уменьши ожидание** — проверяй коллизии до согласования, а не после
4. **Развивай** — автоматизируй проверку коллизий

## Связь с DevOps

Конвейер инноваций = DevOps для инженерных проектов:
- **CI (непрерывная интеграция):** каждое новое данное (бурение, ГТМ, ГДИС) обновляет модель
- **CD (непрерывная поставка):** каждый обоснованный проект проходит полный цикл до внедрения
- **Мониторинг:** результаты пилотов отслеживаются и сравниваются с прогнозом
